Implementation Plan

Cloudflare SEO Memwiki: GitOps to Live Publishing

A comprehensive plan for starting with reviewable GitOps static publishing, then evolving into R2-backed API publishing without changing Memwiki's canonical content model.

Prepared for the Memwiki project. Cloudflare platform references checked May 28, 2026.

Thesis

Build the first production integration around a conservative GitOps workflow, but make every interface compatible with live publishing from day one. The public contract should be "submit sources, produce drafts, validate, promote, publish artifacts." Whether promotion creates a pull request or writes directly to R2 is an implementation detail behind a publisher interface.

This avoids premature operational complexity while preserving the path to near-real-time API-driven content updates.

Target Architecture

The system has four logical layers:

  1. Memwiki core: source registration, extraction, compilation, linting, promotion, claim provenance, and static export.
  2. Publisher adapters: GitOps publisher first, R2 live publisher second, with a shared publish manifest.
  3. Cloudflare runtime: Workers Static Assets for the public site and API routes, plus bindings to R2, KV, D1, and optional Vectorize.
  4. SEO intelligence: content quality checks, internal link graph, structured data validation, redirect/canonical policy, freshness scoring, and content gap reports.
Capability Phase 1 GitOps Phase 2 R2 live publishing
Canonical content HTML files committed to Git. Same HTML files stored as versioned R2 objects, with Git remaining an optional mirror.
Review Pull request review. Draft approval through API, with optional PR mirroring for high-risk changes.
Publish latency Deployment-bound. API-bound, followed by targeted cache purge.
Rollback Revert commit or redeploy previous build. Restore previous R2 object version and index manifest.
SEO output Fully static crawlable HTML. Fully crawlable HTML served from static assets or R2 through the Worker.

Publisher Contract

Add a publisher abstraction to Memwiki before adding any Cloudflare-specific behavior. The important constraint is that the publish artifact format must be identical across GitOps and R2.

class Publisher(Protocol):
    def publish(self, manifest: PublishManifest) -> PublishResult:
        ...

class PublishManifest(TypedDict):
    publish_id: str
    workspace_id: str
    promoted_draft_id: str
    changed_pages: list[PageArtifact]
    changed_assets: list[AssetArtifact]
    changed_indexes: list[IndexArtifact]
    redirects: list[RedirectRule]
    cache_urls: list[str]
    event_log_path: str
    generated_at: str

Publisher adapters should never compile content themselves. They only receive already-validated promoted artifacts and place them into the target system.

Phase 1: GitOps Static Publishing

Goal

Make Memwiki capable of updating a Cloudflare static site through reviewable repository changes. API clients and coding agents can submit source material, but public content changes still pass through Git review.

Implementation Steps

  1. Add memwiki export cloudflare or a library equivalent that exports static site artifacts into a configured site output directory.
  2. Generate public pages from canonical wiki/ HTML, preserving JSON-LD and adding site chrome only at export time.
  3. Generate sitemap.xml, robots.txt, RSS or Atom if relevant, search index JSON, entity index JSON, and related-link data.
  4. Generate Cloudflare-compatible _headers and redirect artifacts where appropriate. Classic Pages supports _headers and _redirects; Workers Static Assets projects can encode equivalent behavior in Worker code when needed.
  5. Add a GitOps publisher that writes exported artifacts into a branch, commits them, and opens a pull request.
  6. Configure Cloudflare deployment from the repository branch or merged output.
  7. Require validation checks in CI before merge: HTML parse, JSON-LD parse, internal links, canonical URLs, duplicate titles, missing descriptions, orphan pages, redirects, and sitemap consistency.

Cloudflare Runtime

Deploy the static site with Workers Static Assets so the same project can later host API routes. The Worker should serve static assets directly except for reserved API paths such as /api/memwiki/*. Cloudflare documents this static-asset-plus-Worker pattern as a single deployment unit.

API Behavior

The first API should be intentionally narrow:

POST /api/memwiki/sources
POST /api/memwiki/drafts
POST /api/memwiki/drafts/:id/check
POST /api/memwiki/drafts/:id/propose
GET  /api/memwiki/proposals/:id

propose should create a Git branch and pull request, not publish directly.

Acceptance Criteria

  • A new source can produce a validated draft and pull request without modifying canonical public content directly.
  • The exported site is fully crawlable without client-side rendering.
  • Every public page has a canonical URL, title, description, stable slug, and JSON-LD where appropriate.
  • Changed pages are visible in the pull request as normal HTML diffs.
  • CI blocks broken internal links, missing provenance, invalid JSON-LD, and sitemap drift.

Phase 1.5: Shared Publish Manifests and Preview Deployments

Goal

Introduce the operational primitives needed for R2 publishing while still using GitOps as the source of publication.

Implementation Steps

  1. Create publish-manifest.json for every export. Include changed artifacts, affected URLs, redirects, index versions, content hashes, and rollback references.
  2. Create immutable content hashes for each exported page and asset.
  3. Add preview exports so reviewers can inspect the exact rendered output before merge.
  4. Store preview bundles in R2 or as CI artifacts, but keep promotion tied to Git merge.
  5. Add a dry-run publisher that returns the exact operations it would perform for GitOps or R2.
  6. Start writing event records in a shape that can be copied into D1 later.

Acceptance Criteria

  • Every export has a machine-readable manifest that can drive either Git commits or R2 object writes.
  • Preview output and merge output are generated by the same exporter.
  • The system can compute affected cache URLs before any live publishing exists.

Phase 2: R2-Backed Live Publishing

Goal

Allow trusted API clients to promote validated drafts directly to a live Cloudflare-backed content store without a full site redeployment.

Storage Model

Store Use
R2 Canonical generated HTML, raw source archives, draft bundles, export bundles, previous versions, and media.
D1 Page registry, current slug-to-object mapping, source records, claim records, publish events, review state, redirects, and rollback pointers.
KV Hot path reads: navigation, sitemap fragments, redirect maps, page summaries, current index pointer, and cache-control hints.
Vectorize Optional semantic search, related content recommendations, and content gap analysis.

R2 Object Layout

raw/<source_id>/source.bin
raw/<source_id>/metadata.json
drafts/<draft_id>/bundle.json
drafts/<draft_id>/pages/*.html
published/pages/<slug>/index.html
published/assets/<hash>.<ext>
published/indexes/search.json
published/indexes/graph.json
published/sitemap.xml
versions/<publish_id>/manifest.json
versions/<publish_id>/pages/*.html

Worker Request Flow

  1. If the request path is under /api/memwiki/, run the authenticated API handler.
  2. If the request matches a deployed static asset, serve it through the assets binding.
  3. If the path maps to a live-published page in D1 or KV, fetch the HTML object from R2 and return it with appropriate cache headers.
  4. If no mapping exists, serve the static 404 page or a configured fallback.

Live Promotion Flow

  1. API client submits source or draft promotion request.
  2. Worker validates auth, payload size, workspace policy, and mutation permission.
  3. Worker enqueues or calls a compiler job. If compilation remains outside Workers, the Worker stores the source and records a pending job for an external agent.
  4. Memwiki compiles and validates the draft using the same checks as GitOps.
  5. Promotion writes new page objects to versioned R2 keys first.
  6. D1 registry updates occur in a transaction-like sequence: create publish event, update page pointers, update redirects, update current index pointer.
  7. KV hot indexes are updated after D1 succeeds.
  8. Cloudflare cache is purged for affected public URLs. Cloudflare supports URL-based cache purging through the cache purge API.
  9. The event log records the promotion, affected URLs, actor, previous version, and rollback pointer.

Acceptance Criteria

  • A validated draft can become public without a redeploy.
  • Published pages remain crawlable HTML responses, not client-rendered placeholders.
  • Each public page can be traced to source IDs, claim IDs, actor, publish ID, and previous version.
  • Rollback restores the previous HTML and index pointers.
  • Cache invalidation is limited to affected URLs, not a full-zone purge by default.

SEO Validation Layer

Add a dedicated validation profile for public website exports. It should run in both GitOps and live publishing.

Validation Rule Failure severity
Title One title per page, unique within the site, within configured length guidance. Block duplicate or missing titles; warn on length.
Description One meta description per indexable page, unique enough to avoid thin duplicates. Block missing descriptions; warn on weak duplicates.
Canonical Every indexable page has one canonical URL matching the publish route. Block.
JSON-LD Structured data parses and references the same canonical entity as the page. Block invalid JSON; warn on missing optional properties.
Internal links All internal links resolve or have explicit redirect records. Block broken links.
Provenance Substantive claims retain source locators and review status. Block accepted claims without provenance.
Freshness Pages with aging sources get a stale score and refresh recommendation. Warn, except for policy-controlled evergreen pages.
Indexability No accidental noindex, blocked robots path, or canonical mismatch on public pages. Block.

Security and Governance

  • Use scoped API tokens or signed requests for all write endpoints.
  • Separate roles for source submission, draft creation, promotion, rollback, and configuration changes.
  • Keep remote model execution opt-in and visible in event logs.
  • Run MIME type checks, file size limits, extension allowlists, and safe metadata extraction on source uploads.
  • Require review for high-risk changes, including homepage updates, money pages, legal/medical/financial claims, redirects, and canonical URL changes.
  • Preserve immutable source objects and previous published versions for rollback and audit.
  • Log actor, IP-derived request metadata where appropriate, user agent, source IDs, draft IDs, and affected URLs for every mutation.

Coding Agent Workflows

The API should be designed for coding agents as first-class clients. Agents need structured outputs, dry-run modes, and clear next actions.

Agent task Required affordance
Assess integration options for a site Expose capabilities, publish mode, storage bindings, and validation policy.
Submit content update Dry-run source ingest and draft generation before mutation.
Review draft quality Return structured lint results with file, object ID, severity, and suggested fix.
Promote content Check-only promotion endpoint and explicit approval token or role.
Debug stale public page Resolve route to publish ID, R2 object key, cache state, source claims, and previous version.

Test Plan

Memwiki Core Tests

  • Publish manifest generation is deterministic.
  • GitOps publisher stages only expected artifacts.
  • SEO validator catches duplicate titles, broken canonical URLs, invalid JSON-LD, and orphan pages.
  • Public export preserves claim IDs and source references.

Cloudflare Worker Tests

  • Static assets are served without invoking mutation logic.
  • API routes reject unauthenticated and under-scoped requests.
  • R2 page fallback returns valid HTML and correct cache headers.
  • Rollback restores page pointers and previous content.
  • Cache purge payload contains only affected URLs.

End-to-End Scenarios

  1. Submit a source through the API, create a draft, open a GitOps pull request, merge it, and verify the deployed static page.
  2. Promote a low-risk draft directly to R2, fetch the public page, and verify provenance and JSON-LD.
  3. Publish a redirect change and verify old and new URLs behave correctly.
  4. Break an internal link intentionally and confirm promotion is blocked.
  5. Rollback a live publish and confirm page HTML, sitemap pointer, and search index pointer return to the previous version.

Milestones

Milestone 1: Static SEO Export

Deliver public-ready HTML export with sitemap, robots, JSON-LD, canonical URLs, internal link data, and SEO linting.

Milestone 2: GitOps Publisher

Deliver branch/commit/PR publishing with CI checks and Cloudflare deployment integration.

Milestone 3: Publish Manifest Contract

Deliver shared publish manifests, preview bundles, affected URL computation, and dry-run publishing.

Milestone 4: Cloudflare Worker API

Deliver authenticated source, draft, check, proposal, and read endpoints on Workers Static Assets.

Milestone 5: R2 Live Publishing

Deliver direct promotion to R2, D1 registry updates, KV hot indexes, targeted cache purge, and rollback.

Milestone 6: SEO Intelligence Loop

Deliver stale content detection, internal linking recommendations, content gap reports, and optional Vectorize-powered semantic discovery.

Risks and Design Constraints

Risk Mitigation
Live API publishing bypasses editorial review. Keep GitOps as default, require role-scoped promotion, and force PR review for high-risk page classes.
Search engines see stale content after live publish. Compute affected URLs in the publish manifest and purge those URLs after successful pointer updates.
R2 object state diverges from Git state. Use publish manifests, content hashes, and optional Git mirroring for live publishes.
Generated SEO pages become thin or duplicative. Block pages without source-backed claims, unique intent, internal links, and sufficient content quality score.
Edge rewriting hides important content from crawlers or debuggers. Keep SEO-critical content in generated HTML. Use HTMLRewriter only for supplemental panels or metadata that can be verified.
